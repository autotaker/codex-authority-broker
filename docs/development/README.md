# 開発プロセス

この文書を、本リポジトリにおける開発プロセスの正本とする。`AGENTS.md` は作業経路と不変条件を示す短い契約であり、タスクファイルと `backlog.json` はタスクの契約、状態、証跡であって、開発プロセスの定義ではない。

## 目的

開発プロセスは、プロダクトの品質、安全性、開発速度を高めるためにある。変更のリスクに応じた統制を適用し、儀式自体を維持するのではなく、実際に得られた結果をもとにプロセスを改善する。

変更経路はファイル形式ではなく、変更がもたらす影響によって分類する。

分類に合理的な疑義がある場合は、プロダクト・安全性契約変更の経路を使用する。特に、受入判定、安全性に関する主張、タスクの完了判定について、承認済みの判断を忠実に記録するだけでなく、その意味を変更する場合は軽量な保守として扱わない。

## 変更経路

### プロダクト・安全性契約変更

PLAN、DEV、独立REVIEW、QAの各ゲートを使用する。DEV、レビュー担当、QA担当の役割は分離する。

この経路には、実行時の振る舞いの変更、および認可、権限、秘密情報の取扱い、fail-closed動作、特権操作手順、またはそれらの受入基準に対する変更を含む。変更対象がMarkdownや設定ファイルだけであっても同様である。

### 文書・証跡の保守

直接編集し、関係する検査だけを実行する。この作業だけを理由として、TASK、PLAN、QA_PLAN、REVIEW_RESULT、QA_RESULTを作成しない。

この経路には、文言、リンク、規範ではない運用案内、状態の整合、メトリクス、既存証跡の忠実な記録または分類を含む。振る舞い、安全性要件、受入基準を変更する場合は、プロダクト・安全性契約変更の経路へ切り替える。

### プロセス改善

プロセス改善には、後述する見直しサイクルを適用する。プロセス改善は独立した開発タスクではなく、TASK、PLAN、DEV、QAの成果物を必要としない。

### 読み取り専用作業

リポジトリや外部の状態を変更しない調査、説明、診断には、開発ゲートを適用しない。

## 5タスクごとのプロセス見直し

前回のプロセス見直し以降に5件の開発タスクが `completed` になった時点で、開発プロセスを見直す。

- 完了したプロダクト・安全性契約変更のタスクを数える。
- 文書保守、証跡保守、終了または後続タスクに置き換えられたタスク、プロセス見直し自体は数えない。
- 5件目のタスクを完了するとプロセス見直しが発生する。次の開発タスクを開始する前に見直し結果を記録する。
- 重大なプロセス上の欠陥が判明した場合は、5件目を待たず直ちに見直す。これには、安全でない動作、品質低下の反復、無効なゲート判定、開発を実質的に阻害する状況について、実害または信頼できるリスクがある場合を含む。前倒しで見直した場合、完了タスク数を0に戻す。

プロセス見直しでは、次の作業だけを行う。

1. 観測した問題、修正案、期待する効果を簡潔に記載した提案パッチを作成する。
2. 提案者以外の者が、最終的な規範変更の差分について独立した整合性レビューを行う。
3. レビューのPASS後、レビュー済みの規範変更だけを適用し、結果を `PROCESS_REVIEW_LOG.md` に記録する。

整合性レビューでは、役割分離、Git操作権限、秘密情報の取扱い、安全性要件、変更経路の分類、関連するプロセス文書との矛盾を確認する。これはプロダクト変更に対する通常のREVIEWやQAゲートではない。

整合性レビュー後に規範を変更した場合は、再度レビューを行う。レビュー結果やその他の非規範的な管理情報を記録するだけの場合は、再レビューを必要としない。

各見直しでは、重複した規則、効果のない成果物、リスクに見合わないゲートを削除または簡略化する。観測されたリスクや欠陥に対して有効な統制は維持または強化する。

## 不変条件

- 子エージェントは、ステージ、コミット、マージ、および `.git` への書き込みを行わない。
- Git操作とリリース公開はメインエージェントが担当する。
- 秘密情報、TOTP値、GitHub Appキー、installation tokenをコミットまたはログへ記録しない。
